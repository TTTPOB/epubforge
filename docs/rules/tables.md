# 表格规则

## 目标

表格修复优先处理的是“结构正确性”，不是美观微调。

对表格来说，最重要的是三件事：

1. 行列关系不能失真
2. 跨页延续不能被误当成新表
3. 修复不能凭空发明单元格内容

## 总原则

### 1. 先保证拓扑，再谈排版

如果一张表的列宽逻辑、`rowspan` / `colspan` 继承、跨页续接关系都不稳定，那么任何局部文字润色都没有意义。scanner 应优先识别结构问题，fixer 优先修结构。

### 2. 单张逻辑表应有单一结构解释

一张逻辑上的连续表格，不应因为 PDF 翻页就被编辑层解释成两套互相冲突的结构。

但反过来，也不能因为相邻两张表主题相近，就轻率把它们合并成一张。

### 3. 合并只发生在证据足够时

表格跨页延续时，常见现象是多段 `<tbody>`、续页第一行半空、前一页末行被截断。只有当这些迹象能共同说明“这是同一行或同一张表的续接”时，才应该合并。

### 4. `rowspan` / `colspan` 是合法结构，不是缺列

很多“短行”其实是继承了上一行的 `rowspan`。不能见到列数不齐就直接补格或改写。

## 规则

### 单一逻辑表的主体结构

如果表格只是因抽取拼接出现多个 `<tbody>`，但逻辑上仍是一张表，应优先整理为单一连续主体。这样做的目的不是追求 HTML 纯洁，而是减少后续误判和渲染歧义。

### 分裂行只在可重建时合并

当跨页导致一行被切成两半时，通常具备这些信号：

- 续页第一行有大量空单元格
- 上一段末行文本明显截断
- 两半拼接后得到完整字段，而不是生造内容

只有满足这些条件，才适合把它视为 split row 并重建。

### 列数不匹配要先判断“继承”还是“损坏”

列数不匹配有两种常见来源：

- 合法的 `rowspan` / `colspan`
- 头部、汇总行或损坏提取造成的真实失衡

如果一行的逻辑宽度在考虑继承单元格后仍不成立，才是真问题。

### 表题、表注和表体要互相印证

一张表的主题和编号通常出现在 `table_title` 或 `caption`。当表体结构不确定时，这些外围文本是判断“同一张表是否在续接”的重要证据。

## 常见模式

### 模式 A: double tbody

表现：

- 同一 `Table.html` 里出现多个 `<tbody>`

这通常意味着跨页续接，而不一定意味着数据错误。若前后主体明显属于同一表，应整理成单一连续结构。

### 模式 B: split row suspected

表现：

- 第二段主体的首行多数为空
- 前一段末行内容被截断

这通常不是“空白数据行”，而是一条逻辑数据行被分页切开。

### 模式 C: column count mismatch

表现：

- 某些行的逻辑列数与其他行不一致

先检查是否是 `rowspan` 继承，其次再看是否是汇总行、标题行或确实损坏。

### 模式 D: header / summary geometry mismatch

表现：

- 头部定义的列数与汇总行、总数行、百分比行对不上

这类问题往往不是正文数据错，而是头部少了占位列或摘要行少了结构信息。

## 允许的修复

- 清理纯粹由续页拼接带来的多余 `<tbody>`
- 在证据充分时合并 split row
- 修正明显失衡的表头或汇总行几何关系
- 统一表题、表注与表体的结构表达

## 不允许的修复

- 凭主观猜测补写缺失单元格内容
- 只因为“渲染更整齐”就抹平真实的 `rowspan` / `colspan`
- 在没有证据时把相邻两张表合并
- 把表格问题降级成纯文本替换，绕过结构判断

## 何时升级为 reviewer

- 同一处既像 split row，又可能是作者原本留空
- 续页前后主题相似，但无法确认是否同一逻辑表
- 头部与汇总行冲突，需要决定是补占位列还是保持原始布局

## 与 memory 的关系

如果一本书的表题格式、续页标识或汇总行样式反复出现，scanner 应把它上升为 convention 或 pattern，而不是每张表单独拍脑袋。
