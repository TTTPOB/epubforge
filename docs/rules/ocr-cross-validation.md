# OCR 双源交叉验证规则

## 目标

传统 Docling+RapidOCR 管线（以下简称 Standard）是文本的主要来源，提供完整的字符串和 Docling JSON layout 结构。Granite Docling VLM 管线（以下简称 Granite）提供 markdown 和 DocTags 结构标签，作为第二证据来源，而非替代。

两条管线都有系统性失效模式——Standard 会漏行、Granite 会在字符识别上引入高错误率汉字。因此，agent 在检查任何文本跨度（段落、标题、脚注、列表项）时，不得单独依赖任何一条管线。

当两个来源对同一位置的输出不一致时，以**语义在上下文中能否成立**作为最终判据；若两个来源的输出都无法语义解释通，不得猜测修复，须上报。

## 总原则

### 1. Granite 是第二证据，不是替代

Standard 管线给出文本和 Docling JSON layout；Granite 给出 markdown + DocTags 结构标签。两者都可能在字符识别和结构标注上出错。

使用方式：把 Granite 输出当成"另一双眼睛"，主要用于：

- **(a) 校对** Standard 的字符错误，特别是形近字、偏旁混淆；
- **(b) 兜底** Standard 偶发的整行/整句缺失；
- **(c) 复核** layout 结构标签（heading/footnote/list 等）。

不要因为 Granite 是 VLM 就盲信其字符识别——Granite 在汉字识别上的字符错误率高于 Standard，且倾向于把 OCR 不确定字符塞入段落而非留空。不要因为 Standard 的 layout 是 Docling 模型给出就盲信其结构标签——见原则 5。

### 2. 文字对照：警惕形近字与系统性 OCR 错误

以下是从实测中发现的典型错误模式，这些模式在两条管线中都可能出现，但不一定同时出现：

**形近偏旁混淆**（任一管线均可能犯）：

- 慮/感、漫澱/漫漫、老鸭/老龄、保钥/保钓
- 驯/瞥、属/犀、薄暄/薄蕎/薄暮、蘊/葛
- 谱/谵、蔡/禁、厦/阈

**「入/人」系统性混淆**（Standard 高频；Granite 偶发同样错误）：

- 考人 → 考入、步人 → 步入、走人 → 走入
- 迈人 → 迈入、人 90 岁 → 入 90 岁

**专有名词形近替换**：人名、地名（赵刚、葛文德、福柯、闽台、琉璃瓦等）对形近字的容忍度极低，须单独复核。

**数字截断**：年份（如 1986 被截成 196）、页码、人口数据。

判定方法：

1. **看上下文**：替换字在该语境下是否是常用搭配？「漫澱余生」不是中文成语，「漫漫余生」是。
2. **跨管线对照**：另一条管线在同一位置输出什么字？哪个版本在汉语中构成合法搭配？
3. **不依赖 majority vote**：两条管线一致错时（如「考人/步人」），仍需用语义判定，一致不等于正确。

### 3. 一致也不等于正确：必须能从语义角度解释通

双源一致是必要条件，不是充分条件。**最终判据是"整段语义能否成立"**。

- 如果两条管线一致输出某词，但该词在该句中明显不通（如「人 90 岁」），仍按错误处理。
- 如果某词单看像错（生僻词、罕见搭配），但通读整段语义自洽，则按正确处理，不强行"纠正"。
- 涉及人名、地名、专有名词时，优先信赖语义一致性而非字形相似度。

### 4. 整行/整句缺失只能由 Granite 兜底

Standard 管线在样本测试中至少出现 2 次「整句被跳过」的现象——属于行级丢失，不是字符错。这类丢失在 Standard 自身文本中无法自我发现。

检测方法：通读 Standard 段落，若**句子主干断裂**（如主语后突然接转折，或上下两句逻辑跳跃），先在 Granite 同页输出中查找是否有补全内容。

反向不成立：Granite 有时会把 OCR 不确定的字符塞进段落，导致段落看起来更长但内容是错的。Granite 多出来的内容必须经过 Standard 验证才能采纳——若 Standard 对应位置完全空缺，且 Granite 输出语义通顺，才可作为候选补全，仍须标注来源。

### 5. layout 结果也不一定准确：双源都要复核

两条管线的结构标签（heading / paragraph / footnote / list / page_header / page_footer）都可能出错。已知模式：

**标题被错切**：Standard 偶尔把多字标题切成两个独立 heading（如把「序一 薄暮时分留夕照」拆成「序一」+「薄暮时分留夕照」），或在标题中插入多余字符（如「二 走进养老院」变成「二〇走进养老院」）。

**节号丢失**：原书中独立成行的阿拉伯数字（「1」「2」「3」）作为节标题时，两条管线都会把它当作普通正文行，没有任何 heading 标记。这种结构信息无法从 OCR 文本自动恢复，必须依赖上下文规则补回（如：序章/序二中独立成行的小数字 + 后续段落起始 → 视为节标题候选）。

**页眉页脚误判**：两条管线对页眉页脚识别总体较稳，但偶尔会把首段第一行误标为页眉。若 page_header 内容超过约 30 字、或语义是叙事内容而非书名/页码，按正文处理。

**脚注归属**：Standard 的 `full.json` 里 footnote 标签可信度高，但其 markdown 中可能降级为段落；Granite 的 `<footnote>` 标签存在，但偶有内容截断。两边对照可恢复脚注本体，见 `footnotes.md`。

判定流程：

1. Standard layout 标签与 Granite DocTags **一致**时：优先采信，但仍按原则 3 做语义复核。
2. **不一致**时：以语义和原书惯例判定，不要默认相信任何一边。
3. **两边都把某结构当成正文段落**（如序二的「1/2/3」节号）：要主动识别这种"双盲漏标"模式，必要时让 doctor/scanner 报告请人工确认，不得自行发明结构。

### 6. 不修复就不要改

跨源对照后**仍无法语义解释通**的字符或结构，不要猜测，不要随机选一边的输出直接写入。

正确处理：在 doctor 报告中标注，让人工或更高 capacity 的 agent 决定。

改写一段含义不明的文本，比保留原始 OCR 错误更有害——后者至少可以被未来流程检测到；前者会污染下游所有快照，且无法追溯来源。

## 常见模式

### 模式 A: 两管线字符不一致，其中之一语义成立

表现：

- Standard 输出「漫澱余生」，Granite 输出「漫漫余生」
- 或 Standard 输出「考人历程」，Granite 输出「考入历程」

处理：

1. 判断哪个版本构成汉语合法搭配
2. 采信语义成立的版本
3. 若对专有名词或稀有词不确定，在 doctor 报告中记录为待人工复核

### 模式 B: 两管线一致，但语义不通

表现：

- 两条管线都输出「人 90 岁」、「步人养老院」
- 通读整段后明显应为「入 90 岁」、「步入养老院」

处理：

- 不能因为两边一致就放过，仍按错误处理
- 以语义判断修正，并在改动记录中说明是双源一致但语义异常

### 模式 C: Standard 句子主干断裂，Granite 有补全

表现：

- Standard 段落：主语出现后下一句突然是转折，找不到谓语或宾语
- Granite 同页位置有一句话填补了这个逻辑空缺

处理：

1. 在 Granite 输出中定位补全句，确认其语义与上下文衔接
2. 若衔接合理，将补全内容标注来源后纳入 Standard 文本
3. 若不确定，不采纳，以 doctor 报告标注

### 模式 D: 结构标签冲突

表现：

- Standard 把某行标为 heading，Granite 标为 paragraph（或反之）
- 两边都把原书节号「1」「2」「3」标为正文

处理：

- 对照原书排版逻辑和上下文判定
- 对「双盲漏标」的节号，生成 doctor 告警，标注候选节标题，等待人工确认
- 不要把结构问题降级成文本替换来回避判断

## 允许的修复

- 在语义判据下明确选择其中一条管线的字符输出
- 采纳 Granite 兜底的缺失句（需标注来源，需语义验证）
- 在两边结构一致且语义成立时采信该结构标签
- 在 doctor 报告中标注跨源不一致或双源存疑的位置

## 不允许的修复

- 因 Granite 是 VLM 就优先采信 Granite 字符输出，跳过语义验证
- 因两条管线输出一致就直接采信，不做语义复核
- 在两边都无法解释通时猜测一个版本写入
- 把 Granite 多出来的内容未经验证直接插入 Standard 文本
- 把结构不确定的位置随意标为 heading 或 paragraph 以消除告警

## 何时升级为 reviewer

- 跨源对照后两个版本都语义成立，无法从上下文判断哪个正确
- 专有名词（人名、地名）在两条管线中各有一个形近版本，都合理
- 疑似「整句缺失」但 Granite 补全内容本身也存在字符错误
- 节号「双盲漏标」需要确认原书结构意图才能补回
- Standard 和 Granite layout 冲突且上下文无法提供足够证据
