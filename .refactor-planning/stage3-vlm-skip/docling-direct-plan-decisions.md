## Skip-VLM 需要你拍板的点

下面只保留我认为无法替你越权决定、但会直接影响计划范围和实现策略的事项。每项都附了建议默认值，方便你回来后直接拍板。

### 1. skip-VLM v1 的目标到底是什么

问题：

- 如果能力预检发现 Docling 不能稳定给出普通脚注 `callout`、跨页 continuation、或表格 `continuation`，skip-VLM v1 还要不要上线？

建议默认值：

- `兼容优先`
- 也就是：只有当关键 Stage 3 contract 基本成立时才上线 skip-VLM；如果预检失败，就不把它当成可用路径。

备选：

- `允许降级`
- 也就是：即使 contract 不完整，也允许作为显式 lossy / 人工复核模式上线。

影响：

- 这会决定计划里 continuation、脚注、表格信号是“必须补齐”还是“允许降级”。

### 2. 是否允许引入有限的 deterministic heuristics

问题：

- 为了补回 Docling 直出里缺失的语义，是否允许使用有限的、非模型式的 deterministic heuristics，例如：
  - 页内拆分脚注归并
  - 表下注释 / 资料来源的邻近恢复
  - continuation 的页对页后处理

建议默认值：

- `允许有限启发式`
- 限定在 deterministic、可审计、可测试的规则内，不引入新的 LLM/VLM 调用。

备选：

- `完全不做启发式`
- 那么计划里就必须明确接受相应语义损失，或在预检失败时直接终止方案。

影响：

- 这会直接改变表格、脚注、续接三类 contract 的实现方式和可达成质量。

### 3. 产物隔离机制具体选哪一种

已定：

- 走“产物隔离”，不走“清理重写”。

还需拍板的问题：

- 隔离机制具体怎么落：
  - `独立命名空间/子目录`
  - `active manifest 指向当前活跃产物`
  - `两者同时使用`

建议默认值：

- `两者同时使用`
- 即：产物按命名空间隔离，Stage 4 再通过 manifest 只读取当前 active set。

影响：

- 这会决定 assemble 如何定位 active Stage 3 产物，以及 mode / `--pages` / unit 数变化时如何拒绝混用旧产物。

### 4. skip-VLM 的开关只做 runtime，还是进入 config/env

问题：

- v1 是否只做 CLI / runtime 参数，还是同步进入 `config.py` 和 env，变成持久化配置？

建议默认值：

- `先只做 runtime / CLI`

备选：

- `同时支持 config/env`

影响：

- 这会影响默认行为稳定性、排障成本，以及计划里需要补多少配置/文档/优先级规则。

### 5. 下游语义债现在还，还是先记账

问题：

- 是否在 v1 一并偿还两笔下游语义债：
  - provenance 能真实表示 `docling`
  - Stage 5 改成按 `image_ref` 而不是页内排序绑定图片

建议默认值：

- `先记账，不阻塞 v1`
- 但必须在计划里明确写成接受中的技术债，不能写成“已兼容”。

备选：

- `现在一起还`

影响：

- 这会决定 v1 范围是否继续扩大到 provenance 和 build 端绑定逻辑。
